Lead Intake

An embeddable, per-organization lead-capture form. An org drops the form onto its own website; a public visitor submits their move details; the submission becomes a project in the org's workspace and the visitor is invited to complete a self-survey for a faster quote.

1. How it works

  1. An admin opens Settings → Embeddable Form, configures the form, and copies an embed snippet (an <iframe> or a small JS widget).
  2. The snippet renders the hosted form on the org's own site. The form is fully public — no login, no auth handshake.
  3. A visitor submits their contact and move details. The request is validated and posted to a public submission endpoint.
  4. The organization is derived server-side from the stored form record, never read from the client. This is the trust boundary: the public formId identifies the form, and the form record owns the org.
  5. The submission is recorded and turned into a project in that org's workspace, along with a customer self-survey upload link.
  6. The visitor sees a success message with a call-to-action to complete the self-survey — a button on mobile, a QR code on desktop.

Origin control: each form carries an allow-list of domains permitted to post to it, so a form can only be submitted from the sites the admin authorizes.

4. Types & contracts

These are the feature's canonical types. Import them — do not redeclare a parallel shape. Each one lives next to the thing it describes (model, schema, or function); this section is the map so you don't have to hunt for them.

Rules for extending the type surface
  • The submission payload's single source of truth is the Zod schema leadSubmissionSchema. Get its type via z.infer (exported as LeadSubmissionInput). To add or change a field, edit the schema — never hand-write a second interface that mirrors it.
  • The I… interfaces are the database document shapes, defined alongside their Mongoose schema. Extend or import them; don't fork a copy (the DB silently drops fields you persist without declaring — a forked type hides that).
  • organizationId is always derived server-side from the form record. Never add a client-supplied org field to any of these types.
  • Component-level draft/state types stay colocated with their component, but some intentionally mirror a canonical shape (a form draft vs. the submission, a settings draft vs. the record). They're catalogued under Component-level types below with what each one shadows — when you add a field to a canonical type, update its shadow in the same change.

Data models

features/lead-intake/models/LeadForm.ts — the stored per-org form record.

interface ILeadForm extends Document {
  organizationId: string;        // Clerk org id — the trusted owner
  formId: string;                // public, unguessable id used in embed URLs
  name: string;
  websiteDomain?: string;        // primary allowed origin host (CORS allow-list)
  allowedDomains: string[];      // additional allowed origin hosts
  theme?: Record<string, unknown>;
  isActive: boolean;
  isDefault: boolean;            // the org's auto-provisioned default form
  createdAt: Date;
  updatedAt: Date;
}

features/lead-intake/models/LeadSubmission.ts — one record per public submission.

interface ILeadSubmission extends Document {
  formId: string;                // the public form id this came through

  data: Record<string, unknown>; // full validated payload
  projectId?: ObjectId | string;
  sourceOrigin?: string;         // request Origin header at submit time
  createdAt: Date;
  updatedAt: Date;
}

Submission payload (the public contract)

features/lead-intake/lib/validators.ts — schema is the source of truth; the type is inferred from it. name + email required, the rest optional; unknown keys stripped.

const leadSubmissionSchema = z.object({
  name:        z.string().trim().min(1).max(200),     // required
  email:       z.string().trim().email().max(200),    // required
  phone:       z.string().trim().max(50).optional(),
  phoneType:   z.string().trim().max(50).optional(),
  origin:      z.string().trim().max(500).optional(),
  destination: z.string().trim().max(500).optional(),
  moveDate:    z.string().trim().max(100).optional(),
  moveSize:    z.string().trim().max(100).optional(),
  notes:       z.string().trim().max(5000).optional(),
}).strip();

type LeadSubmissionInput = z.infer<typeof leadSubmissionSchema>;  // import this; don't redeclare

5. Tests passed

  • Trust boundary holds: a client-supplied organization id is ignored; the org is derived from the stored form.
  • A submission creates a project plus a customer self-survey link in the correct org.
  • The success CTA shows correctly — button on mobile, QR code on desktop.
  • The public form page and the JS widget both render and submit.
  • The settings page auto-provisions a default form and saves configuration scoped to the signed-in org.
  • Allowed domains are normalized, de-duplicated, and capped; submissions from listed origins are accepted and others are rejected.
  • An inactive or unknown form shows “Unavailable” and the submission endpoint returns 404 — no silent project creation.
  • The form renders inside a cross-site iframe with no auth-handshake redirect and no framing block; protected routes remain gated.
  • Org isolation: a signed-in admin cannot modify another org's form.
  • Type-check (tsc --noEmit) is clean; existing settings pages are unaffected.
